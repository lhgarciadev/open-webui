# Etapa 2.3 - Ejecutar Persistencia de Base de Datos

## Prompt
Implementa el despliegue de persistencia de datos de la aplicacion (usuarios, sesiones, chats y metadatos), como extension inmediata despues de la Etapa 2:

1. Definir stack de persistencia para produccion:
   - Base transaccional: PostgreSQL administrado o contenedor dedicado.
   - Cache/sesiones (si aplica): Redis administrado o contenedor dedicado.
2. Configurar variables y secretos de entorno para backend:
   - `DATABASE_URL` apuntando a PostgreSQL persistente.
   - `REDIS_URL` (si se habilita Redis).
   - Credenciales via secretos del proveedor (no hardcodeadas).
3. Garantizar persistencia real en deploy:
   - Si es contenedor propio: volumen persistente para PostgreSQL.
   - Si es servicio administrado: confirmar plan con almacenamiento durable y backups.
4. Ejecutar migraciones de esquema en entorno objetivo:
   - `alembic upgrade head` o mecanismo equivalente del proyecto.
5. Documentar operacion en `planning/`:
   - Arquitectura elegida, variables requeridas, pasos de provisionamiento y rollback.

Entrega:
- Configuracion aplicada (sin exponer secretos).
- Evidencia de migraciones ejecutadas.
- Documento operativo de persistencia y recovery basico.
